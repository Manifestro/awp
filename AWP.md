# Agent Wake Protocol (AWP)

Статус: ранний черновик  
Версия: 0.1-draft

## 1. Назначение

Agent Wake Protocol (AWP) — стандарт доставки внешних событий в неактивную агентную сессию.

AWP нужен в ситуации, когда Codex, Claude Code или другой агент завершил текущую работу и ожидает внешнее событие. Источник события передаёт его через AWP, после чего локальный клиент возобновляет нужную сессию и передаёт событие агенту.

Пример:

1. Агент через Sinores отправляет сообщение в WhatsApp.
2. Агент завершает текущий turn и становится неактивным.
3. Sinores получает новое входящее сообщение.
4. Sinores создаёт AWP-событие.
5. AWP доставляет событие на компьютер пользователя.
6. AWP Client возобновляет нужную сессию Codex или Claude Code.

AWP не определяет, какие действия агент выполнит после пробуждения. Он только доставляет событие и инициирует продолжение сессии.

## 2. Роли

### 2.1. AWP Client

AWP Client устанавливается в среде, где работает агент: на компьютере пользователя, в IDE, рядом с Codex CLI, Claude Code или другим совместимым runtime.

Обязанности клиента:

- устанавливать исходящее соединение с AWP Service;
- регистрировать доступные локальные агентные сессии;
- принимать новые события;
- определять, какую сессию необходимо возобновить;
- передавать событие в Codex, Claude Code или другой агент;
- подтверждать получение и обработку события.

AWP Client не обязан иметь публичный домен, статический IP или открытый входящий порт.

### 2.2. AWP Service

AWP Service — сервис доставки событий между Event Server и AWP Client. По назначению он похож на webhook relay, но поддерживает доставку события клиенту, который находится за NAT, имеет динамический IP или временно отключён.

Обязанности сервиса:

- аутентифицировать Event Server и AWP Client;
- принимать AWP-события;
- определять целевого клиента;
- хранить события до момента доставки;
- доставлять события по исходящему соединению клиента;
- повторять доставку неподтверждённых событий;
- предотвращать повторную обработку одного события.

Основным транспортом для локального клиента предполагается исходящее постоянное соединение, например WebSocket. Для серверных клиентов также может поддерживаться прямой HTTP webhook.

### 2.3. Event Server

Event Server — система, в которой возникают события. Она всегда работает независимо от агентной сессии и создаёт AWP-событие при наступлении нужного условия.

Примеры Event Server:

- Sinores создаёт событие при получении сообщения WhatsApp;
- GitHub-интеграция создаёт событие при новом комментарии в Pull Request;
- почтовый сервер создаёт событие при получении письма;
- система мониторинга создаёт событие при срабатывании алерта;
- планировщик создаёт событие в заданное время.

Event Server не должен знать внутреннее устройство Codex или Claude Code. Он передаёт событие в AWP Service, используя стандартный формат AWP.

## 3. Общий поток

```text
┌──────────────┐       AWP event       ┌─────────────┐
│ Event Server │ ────────────────────▶ │ AWP Service │
│   Sinores    │                       │ queue/relay │
└──────────────┘                       └──────┬──────┘
                                            │
                              outbound connection
                                            │
                                     ┌──────▼─────┐
                                     │ AWP Client │
                                     └──────┬─────┘
                                            │ resume + event
                                  ┌─────────▼─────────┐
                                  │ Codex/Claude Code │
                                  └───────────────────┘
```

AWP Client первым устанавливает соединение с AWP Service. Поэтому доставка работает без публичного адреса на компьютере пользователя.

## 4. Формат события

Минимальный AWP envelope:

```json
{
  "type": "awp",
  "data": {}
}
```

Поля:

- `type` — идентификатор формата. Для AWP-события значение равно `"awp"`.
- `data` — объект с данными события. Его содержимое определяет Event Server.

AWP Service обязан доставить `data` без изменения. AWP не ограничивает пользовательские поля внутри `data`, но их значения должны быть представлены в JSON.

## 5. Пример события Sinores

Sinores может передать идентификатор WhatsApp-канала, номер отправителя и полученное сообщение:

```json
{
  "type": "awp",
  "data": {
    "source": "sinores",
    "event": "message.received",
    "channel_id": "channel_123",
    "from": "+77001234567",
    "message_id": "wa_message_456",
    "text": "Да, давай завтра в 10"
  }
}
```

После получения AWP Client возобновляет связанную с событием сессию и передаёт ей смысл события:

```text
Новое WhatsApp-сообщение через Sinores.
Канал: channel_123
Отправитель: +77001234567
Сообщение: Да, давай завтра в 10
```

## 6. Пользовательские данные

Каждый Event Server может определить собственную структуру `data`.

Например, GitHub-интеграция может отправить:

```json
{
  "type": "awp",
  "data": {
    "source": "github",
    "event": "pull_request.comment.created",
    "repository": "example/project",
    "pull_request": 42,
    "comment_id": 891
  }
}
```

AWP Client не обязан понимать прикладные поля. Его задача — надёжно передать событие в агентную сессию. Интерпретацией `data` занимается агент или установленный адаптер источника.

## 7. Открытые вопросы стандарта

Следующие части необходимо определить в следующих версиях документа:

- как событие указывает целевого пользователя, устройство и сессию;
- как сессия подписывается на определённые события;
- как устроена аутентификация;
- как AWP Client поддерживает соединение с AWP Service;
- как подтверждается доставка;
- как исключается повторная обработка события;
- сколько времени событие хранится, если клиент offline;
- как различаются ручное и автоматическое пробуждение;
- как ограничиваются права автоматически пробуждённой сессии;
- как несколько сообщений объединяются в одно пробуждение.

## 8. Основной принцип

> MCP позволяет активному агенту обращаться к внешним системам. AWP позволяет внешним системам доставлять события неактивному агенту и возобновлять его сессию.
